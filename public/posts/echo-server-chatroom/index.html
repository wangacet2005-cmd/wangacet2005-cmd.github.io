Day 7：TCP Echo Server 完整实战——从 Echo 到聊天室

March 7, 2026 · wangacet2005-cmd

写在前面：这是第 7 天，网络编程三天的收尾。今天把 Day 5 的多线程、Day 6 的 epoll 融合成一个完整项目，补上粘包处理、quit 命令、SIGPIPE 保护，最后再把 Echo Server 升级成多人聊天室。代码从玩具级别走向接近生产级别。

1. 今天新增的两个核心问题#

粘包处理#

TCP 是流协议，没有消息边界。客户端发两条消息，服务端可能一次 recv 全收到，也可能一条都没收完。解决方案是以 \n 作为消息分隔符，收到数据先放进缓冲区，找到 \n 才处理一条完整消息。

每个客户端需要独立缓冲区#

如果多个客户端共用一个 char buf[1024]，fd=5 的 “hel” 会被 fd=6 的数据覆盖，半包数据永久丢失。必须每个 fd 对应一个独立的 string，让数据能跨多次 epoll_wait 保留。

2. 三个关键设计决策#

buffers.erase(fd) 不能忘#

忘了调用有两个后果：

内存泄漏：每个断开的客户端留下残留数据，服务端长期运行内存持续增长。

fd 复用时数据污染：操作系统会把断开的 fd 编号重新分配给新客户端。新客户端连上来，buffers[fd] 里还有上一个客户端的半包数据，两段数据拼在一起，处理出来的消息完全是乱的。这种 bug 复现概率低，极难排查。

客户端发半包后断开#

客户端发 “hel” 后断开，服务端把 “hel” 存入缓冲区，找不到 \n 不处理。recv() 返回 0 触发 disconnect_client，buffers.erase(fd) 把 “hel” 随之清除，不会被错误处理。

为什么要处理 \r#

Windows 的 nc 发送的换行是 \r\n，不处理 \r 的话 “quit\r” != “quit”，quit 命令永远匹配不上。

3. 聊天室核心设计#

两种消息状态#

用 name.empty() 区分，不需要额外的状态标志位：

  • 用户名为空：第一条消息就是用户名，设置后广播上线通知
  • 用户名已设置：正常发消息，广播给所有其他人

广播函数#

遍历所有 fd，跳过发送者自己，跳过还没登录的客户端。

断开时也要广播#

不管是发了 quit 还是直接关闭连接，只要有用户名就广播离线通知，其他用户不会看到有人无声消失。

核心总结#

粘包用缓冲区加分隔符解决，不能假设一次 recv 是完整消息。每个 fd 必须有独立缓冲区，断开时必须 erase，否则 fd 复用后新客户端收到脏数据。SIGPIPE 必须忽略，否则一个客户端异常断开导致整个服务端崩溃。用户名状态用 name.empty() 区分，不需要额外标志位。

记住：buffers.erase(fd) 忘了调用，不只是内存泄漏，更危险的是 fd 被复用后新客户端收到脏数据，这种 bug 极难复现和排查。
